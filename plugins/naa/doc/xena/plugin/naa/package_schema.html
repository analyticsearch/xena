Package Document Format for Xena 2.0

Abstract

This specification documents the rules for the package document format. A package is a collection of distinct XML (extensible markup language) documents bundled together as one XML document instance. XML documents can be designated as metadata about other XML documents within the package instance, thus allowing the content of the package to be self-describing. This specification consists of an XML schema and an explanation of the elements and attributes in that schema.

Status

Not released for external comment. Implemented in Xena 1.1.17 and later versions.

1. Introduction

The package data format bundles arbitrary XML document instances and associated metadata (encoded in a XML vocabulary) into one XML document instance that is suitable for indefinite preservation within a digital repository. This format does not impose any particular document format or metadata scheme upon instances contained within a package; they only need to be well-formed XML. In order to correctly process the arbitrary XML document instances within a package, however, an application will need information about the XML vocabulary used by the instances. It is possible to bundle this information together with the document instance within a package.

The format allows for any number of XML documents to be stored within a package instance. Similarly, any number of metadata profiles can be associated with any one XML document. The format allows contained XML documents to be ordered in a hierarchical fashion, by containing one package instance inside another.

2. Dependencies

The key words 'MUST', 'MUST NOT', 'REQUIRED', 'SHALL', 'SHALL NOT', 'SHOULD', 'SHOULD NOT', 'RECOMMENDED', 'MAY', and 'OPTIONAL' in this document are to be interpreted as described in RFC 2119 (reference 6.2)

References to XML, XML namespace, and XML Schema are to be interpreted according to Extensible markup language (reference 6.3), Namespaces in XML (reference 6.4), XML schema part 0 (reference 6.5), XML schema part 1 (reference 6.7), XML schema part 2 (reference 6.1) respectively.

3. XML namespace

A package instance SHOULD use the namespace declaration:

http://preservation.naa.gov.au/package/2.0.

A package instance MAY use the namespace prefix: package.

4. XML schema

A package instance MUST conform to the following XML schema:

<?xml version="1.0"?>

<xs:schema targetNamespace="http://preservation.naa.gov.au/package/2.0" xmlns:xs="http://www.w3.org/2001/XMLSchema" version="1.0">

<xs:annotation>

<xs:documentation xml:lang="en"> package. A schema to bundle arbitrary XML documents into one XML document instance. Developed by the National Archives of Australia. Copyright 2005 Commonwealth of Australia.

</xs:documentation>

</xs:annotation>

<xs:element name="package">

<xs:complexType>

<xs:all>

<xs:element name="meta" minOccurs="0 maxOccurs="unbounded">

<xs:complexType>

<xs:sequence>

<xs:any namespace="#any" processContents="lax"/>

</xs:sequence>

<xs:attribute name="id" type="xs:ID" use="optional"/>

</xs:complexType>

</xs:element>

<xs:element name="content" minOccurs="0">

<xs:complexType>

<xs:sequence>

<xs:any namespace="##any" processContents="lax" minOccurs="0" maxOccurs="unbounded"/>

</xs:sequence>

<xs:attribute name="id" type="xs:ID" use="optional"/>

</xs:complexType>

</xs:element>

</xs:all>

</xs:complexType>

</xs:element>

</xs:schema>

5. Explanation of elements and attributes

5.1 package element

This is the root element for a package instance. All the other elements in a package instance MUST be contained within this element. It consists of zero or more meta elements and zero or one content elements. The meta elements contain information that is useful to the management and exploitation of the data contained within the content element. The content element contains the data that is the primary focus of preservation. Since the content element can contain arbitrary well-formed XML, it is possible for a package instance to be contained within another package instance.

XML Code

<package>

Example fragment

An empty package instance


<package />

5.2 meta element

This is the container for metadata instances that applies to all the XML-encoded data stored in the content element. Each meta instance contains a specific metadata profile that relates to the data in the content element. Any well-formed XML is allowed: an application that uses the package format must have access to other sources of information in order to respond appropriately to the contents of a meta element. If an application does not have any special processing routines for the content of a meta element it MUST pass the content to any client application as unmodified XML (ie, an application MUST NOT ignore the contents of a meta element simply because it does not understand the meta's XML vocabulary). The use of XML namespaces to identify the contents of a meta element is strongly advised. A meta element MAY have a value associated with its id attribute. This attribute uniquely identifies the metadata profile within the entire package instance.

XML code

<meta>

Example fragment

A meta fragment that contains two metadata profiles. The first profile contains a checksum encoded in the National Archives of Australia's checksum document format and the second contains Dublin Core metadata:

<package:meta>

<checksum:checksum
xmlns:checksum="http://preservation.naa.gov.au/checksum/1.0"

type="http://preservation.naa.gov.au/md5">

3839788fd7525bb4b9ac4834ff041dc6

</checksum:checksum>

<dc:creator xmlns:dc="http://purl.org/dc/elements/1.1">Simon
Davis</dc:creator>

</package:meta>

Attributes

id

5.2.1 id attribute

This provides a unique identifier for a metadata profile within a meta element. The identifier MUST be unique throughout the entire package instance. The identifier MUST be a valid XML ID.

XML code

id

Example fragment

A fragment of a meta element that has the id of "md77":

<meta id="md77">

5.3 content element

This contains a distinct XML document instance that is the subject of preservation. Any well-formed XML is allowed: an application that uses the package format must have access to other sources of information in order to respond appropriately to the contents of a content element. If an application does not have any special processing routines for the content of a content element it MUST pass the content to any client application as unmodified XML (ie, an application MUST NOT ignore the contents of a content element simply because it does not understand its XML vocabulary). The use of XML namespaces to identify the contents of a content element is strongly advised. A content element MAY have a value associated with its id attribute. This attribute uniquely identifies the XML document contained within the content element throughout the entire package instance.

A packageinstance MAY be contained within a content instance. A use case for doing so is to mimic within a package the structure of a directory tree where directories can contain other directories (ie, other package instances contained within content elements) or data files (ie, other, arbitrary XML document instances contained within content elements).

XML code

<content>

Example fragment

A content element that consists of a fragment of a XML instance encoded in the National Archives of Australia's plaintext document format:

<content>

<plaintext:plaintext xmlns:plaintext="http://preservation.naa.gov.au/plaintext/1.0">

<plaintext:line xml:space="preserve">This essay is a brief, practical introduction to</plaintext:line>

<plaintext:line xml:space="preserve">recordkeeping ideas, designs and technologies in </plaintext:line>

<plaintext:line xml:space="preserve">recordkeeping systems.</plaintext:line>

</plaintext:plaintext>

</content>

Attributes

 

5.3.1 id attribute

This provides a unique identifier for a XML document instance within a content element. The identifier MUST be unique throughout the entire package instance. The identifier MUST be a valid XML ID.

XML code

id

Example fragment

A fragment of a content element that has the id of "c6":

<content id="c6">
