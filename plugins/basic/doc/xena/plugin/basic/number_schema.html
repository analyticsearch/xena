Number Type Format

10 October 2002

URI: http://preservation.naa.gov.au/number/1.0

Contributors: Simon Davis, Chris Bitmead, and Andrew Lee

Abstract

This specification documents the rules for the number document format. number represents any data that is included in the following numerical formats (defined by XML schema part 2, reference 7.1):

  • integer

  • float 

  • byte 

  • decimal 

  • double 

  • int 

  • short 

  • unsignedByte 

  • unsignedInt 

  • unsignedLong 

  • unsignedShort

Status

This document has been created as part of the National Archives of Australia's Agency to Researcher Digital Preservation Project. It is a draft and has not yet been quality checked or signed-off by the Project Board. It is subject to change.

The number XML Schema (see section 4) has been checked with the University of Edinburgh-World Wide Web Consortium's XSV (XML Schema Validator) (reference 7.10) with no errors reported.

1. Introduction

1.1 [Definition:] number represents an element which MUST contain that is included in the following numerical formats (defined by XML schema part 2, reference 7.1):

  • integer

  • float 

  • byte 

  • decimal 

  • double 

  • int 

  • short 

  • unsignedByte 

  • unsignedInt 

  • unsignedLong 

  • unsignedShort

1.2 Much data may only be defined by the type of the data rather than the context of the data in the document instance. For simplicity it may be beneficial to represent data by its type for the purpose of manipulation and rendering.

1.3 The number type format is designed to allow the definition of data by its type rather than its context. This format depends heavily on other, already accepted, standards such as Unicode and XML. The format consists of both an XML Schema to define the structure that instances must conform to as well as a set of View requirements that determine how the various elements within that structure should be rendered. These two components--the Schema and the View--can be equated to the concepts of 'data object' and 'information object' in the upcoming ISO Standard, Open Archival Information System Reference Model.

1.4 If data context is significant it is recommended that data be referenced by an explanatory element name with an attribute describing the data type, or that the number type is contained within an contextual element descriptor.

1.5 This specification is intended to define what the number format is and how it is utilised.

2. Dependencies

2.1 The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in RFC 2119 (reference 7.3).

2.2 The terms 'block', and 'visual' in this document are to be interpreted as described in Cascading style sheets, level 2 (reference 7.2).

2.3 References to XML, XML namespaces, XML Schema, and Unicode characters are to be interpreted according to Extensible markup language (reference 7.4), Namespaces in XML (reference 7.5), XML schema part 0 (reference 7.7), XML schema part 1 (reference 7.9), XML schema part 2 (reference 7.1), The Unicode standard: Version 3.0 (reference 7.8) respectively.

3. XML namespace

3.1 A number instance SHOULD use the namespace declaration: http://preservation.naa.gov.au/number/1.0

3.2 A number instance SHOULD use the namespace prefix: number

4. XML Schema

4.1 A number instance MUST conform to the following XML Schema:

<?xml version="1.0" encoding="UTF-16"?>
<xsd:schema targetNamespace="http://preservation.naa.gov.au/number/1.0" xmlns:xsd="http://www.w3.org/2001/XMLSchema" xmlns="http://preservation.naa.gov.au/number/1.0" elementFormDefault="qualified" attributeFormDefault="unqualified">
    <xsd:element name="number">
         <xsd:simpleType>
               <xsd:union memberTypes="xsd:integer xsd:float xsd:byte xsd:decimal xsd:double xsd:int xsd:short xsd:unsignedByte xsd:unsignedInt xsd:unsignedLong xsd:unsignedShort"/>
         </xsd:simpleType>
    </xsd:element>
</xsd:schema>

5. Explanation of elements and attributes

5.1 number element

5.1.1 The only element in a number instance. No other element is allowed within a number instance.

XML code

<number:number>

Rules

The order of  characters within the number element MUST be preserved.

Example fragment

<number:number>1254</number:number>

Attributes

nil

6. Views

6.0.1 A compliant application MUST support the requirements of the number View.

6.0.2 The number View is the default view for the number type format: when a number instance is rendered with no other arguments, it MUST be rendered in the number View.

6.1 Number View

6.1.1 The number View is designed principally for visual devices.

6.1.2 The content of each number element MUST be rendered within its own block.

6.1.3 Empty number elements MUST be rendered.

6.1.4 The View MUST render numbered entities as there associated character.

6.1.5 The View MUST render the content of each element using a monospace font.

6.16 The View MUST render the content of each number element as right justified.

6.1.7 The View MAY wrap number element content to fit the dimensions of the view.
