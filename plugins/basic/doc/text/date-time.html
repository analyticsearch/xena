dateTime Type Format

18 October 2002

URI: http://preservation.naa.gov.au/dateTime/1.0

Contributors: Simon Davis, Chris Bitmead, and Andrew Lee

Abstract

This specification documents the rules for the dateTime document format. dateTime represents any date and/or time data. The dateTime format is an implementation of a subset of the dateTime as described by RFC 8601:2000 E.

Status

This document has been created as part of the National Archives of Australia's Agency to Researcher Digital Preservation Project. It is a draft and has not yet been quality checked or signed-off by the Project Board. It is subject to change.

The dateTime XML Schemas (see section 4) have been checked with the University of Edinburgh-World Wide Web Consortium's XSV (XML Schema Validator) (reference 7.10) with no errors reported.

1. Introduction

1.1 Definitions

1.1.1 General Information

There exists three different types of calendar day:
Calender date: Consisting of calendar year, calendar month and ordinal day number within month.
Ordinal date: Consisting of calendar year and ordinal day number within year
Week date: Consisting of calendar year to which week belongs, calendar week number and ordinal day number within week

Possible date seperators:
- (hyphen, may also be used to indicate omitted components)
: (colon)
/ (solidus)
# (number sign)

Character representations for calendar components:

[Y] - year
[M] - month
[D] - day
[w] - week
[h] - hour
[m] - minute
[s] - second
[n] - digits
[+/-] - positive or negative.

Note: An underlined character represents zero or more digits in the corresponding time element.

1.1.2 Coordinated Universal Time (UTC)

To express a time in UTC, the dateTime must be followed immediately, without spaces, by the letter [Z].
All dateTime should be expressed as either UTC or local time.
To express local time in place of the [Z] a [-] or [+] sign followed the amount of hours and minutes before or after UTC the local time is.

[T] represents the begining of the time portion of datetime

1.1.3 Calender Date:

[+/-]YYYYY-MM-DDThh:mm:ss.n+[Z|+|-]hh:mm

Where:

[Z|+|-] represents "Z" or "-" or "+" as appropriate.

[+/-] represent negative BC or AD with - and + repsectively

The hyphen character (-)  must be used to represent omitted time component.

a date, with no time component would be

YYYYY-MM-DDT-:-:-

a time with no date would be

-----Thh:mm:ss.n+[Z|+|-]hh:mm

The following table is a representation of different dates and times

dateTime Date component Time component Description
-12000-11-23T10:35:12Z -12000-11-23 10:35:12Z 10:35 AM (UTC), 23rd of November 12000 BC
-----T23:35:25.0009-01:30 ----- 23:35:25.0009-01:30  25.0009 seconds past 11:35 PM local time which is 1.5 hours behind UTC
0054-02-27T-:-:- 0054-02-27 -:-:- 27th February 54AD
10000-12--T-:-:- 10000-12-- -:-:- December 10000 AD

 

Ranges for dateTime components in calendar date:

Time Component Minimum Value Maximum Value
[Y] - year 0000 Infinite
[M] - month 01 12
[D] - day 01 31
[w] - week N/A N/A
[h] - hour 00 24
[m] - minute 00 59
[s] - second 00 60
[n] - digits 0 .9r

10000-12T-:-:- 10000-012 -:-:- 12th day of 10000 AD

1.2 Much data may only be defined by the type of the data rather than the context of the data in the document instance. For simplicity it may be beneficial to represent data by its type for the purpose of manipulation and rendering.

1.3 The dateTime type format is designed to allow the definition of data by its type rather than its context. This format depends heavily on other, already accepted, standards such as Unicode and XML. The format consists of both an XML Schema to define the structure that instances must conform to as well as a set of View requirements that determine how the various elements within that structure should be rendered. These two components--the Schema and the View--can be equated to the concepts of 'data object' and 'information object' in the upcoming ISO Standard, Open Archival Information System Reference Model.

1.4 If data context is significant it is recommended that data be referenced by an explanatory element name with an attribute describing the data type, or that the dateTime type instance is contained within an contextual element descriptor.

1.5 This specification is not intended to define what data may or may not be encoded as dateTime. 

1.6 This specification is intended to define what the dateTime format is and how it is utilised.

2. Dependencies

2.1 The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in RFC 2119 (reference 7.3).

2.2 The terms 'block', and 'visual' in this document are to be interpreted as described in Cascading style sheets, level 2 (reference 7.2).

2.3 References to XML, XML namespaces, XML Schema, and Unicode characters are to be interpreted according to Extensible markup language (reference 7.4), Namespaces in XML (reference 7.5), XML schema part 0 (reference 7.7), XML schema part 1 (reference 7.9), XML schema part 2 (reference 7.1), The Unicode standard: Version 3.0 (reference 7.8) respectively,  ISO 8601:2000 E (Reference 7.11).

3. XML namespace

3.1 A dateTime instance SHOULD use the namespace declaration: http://preservation.naa.gov.au/dateTime/1.0

3.2 A dateTime instance SHOULD use the namespace prefix: primitive

4. XML Schema

4.1 A dateTime date instance MUST conform to the following XML Schema:

<?xml version="1.0" encoding="UTF-8"?>
<xsd:schema targetNamespace="http://preservation.naa.gov.au/primitive/1.0" xmlns="http://preservation.naa.gov.au/primitive/1.0" xmlns:xsd="http://www.w3.org/2001/XMLSchema" elementFormDefault="qualified" attributeFormDefault="unqualified">
   <xsd:simpleType name="calenderDate" final="restriction">
      <xsd:restriction base="xsd:dateTime">
         <xsd:pattern value="[0-9]?[0-9][0-9][0-9][0-9]\-[0-9][0-9]\-[0-9][0-9]T[0-9][0-9]:[0-9][0-9]:[0-9][0-9]\.?[0-9]*Z?([0-9][0-9]:[0-9][0-9])?"/>
      </xsd:restriction>
   </xsd:simpleType>
   <xsd:simpleType name="weekDate" final="restriction">
      <xsd:restriction base="xsd:dateTime">
         <xsd:pattern value="[0-9]?[0-9][0-9][0-9][0-9]\-[0-9][0-9]\-[0-9][0-9]T[0-9][0-9]:[0-9][0-9]:[0-9][0-9]\.?[0-9]*Z?([0-9][0-9]:[0-9][0-9])?"/>
      </xsd:restriction>
   </xsd:simpleType>
   <xsd:simpleType name="ordinalDate" final="restriction">
      <xsd:restriction base="xsd:dateTime">
         <xsd:pattern value="[0-9]?[0-9][0-9][0-9][0-9]\-[0-9][0-9][0-9]T[0-9][0-9]:[0-9][0-9]:[0-9][0-9]\.?[0-9]*Z?([0-9][0-9]:[0-9][0-9])?"/>
      </xsd:restriction>
   </xsd:simpleType>
   <xsd:element name="dateTime">
      <xsd:complexType>
         <xsd:choice>
            <xsd:element name="calenderDate" type="calenderDate"/>
            <xsd:element name="weekDate" type="weekDate"/>
            <xsd:element name="ordinalDate" type="weekDate"/>
         </xsd:choice>
      </xsd:complexType>
   </xsd:element>
</xsd:schema>
