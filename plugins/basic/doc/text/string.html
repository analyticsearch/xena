String Type Format

10 October 2002

URI: http://preservation.naa.gov.au/string/1.0

Contributors: Simon Davis, Chris Bitmead, and Andrew Lee

Abstract

This specification documents the rules for the string document format. string represents plain (ie, 'unformatted') text content that is normally encoded in ASCII or some derivative as XML document instances. The format consists of both an XML schema and a set of rendering rules.

Status

This document has been created as part of the National Archives of Australia's Agency to Researcher Digital Preservation Project. It is a draft and has not yet been quality checked or signed-off by the Project Board. It is subject to change.

The string XML Schema (see section 4) has been checked with the University of Edinburgh-World Wide Web Consortium's XSV (XML Schema Validator) (reference 7.10) with no errors reported.

1. Introduction

1.1  [Definition:] The string datatype represents character strings in XML. A character is an atomic unit of communication; it is not further specified except to note that every character has a corresponding Universal Character Set code point, which is an integer.

1.2 Much data may only be defined by the type of the data rather than the context of the data in the document instance. For simplicity it may be beneficial to represent data by its type for the purpose of manipulation and rendering.

1.3 The string type format is designed to allow the definition of data by its type rather than its context. This format depends heavily on other, already accepted, standards such as Unicode and XML. The format consists of both an XML Schema to define the structure that instances must conform to as well as a set of View requirements that determine how the various elements within that structure should be rendered. These two components--the Schema and the View--can be equated to the concepts of 'data object' and 'information object' in the upcoming ISO Standard, Open Archival Information System Reference Model.

1.4 If data context is significant it is recommended that data be referenced by an explanatory element name with an attribute describing the data type, or that the string type is contained within an contextual element descriptor.

1.5 Many data instances may be "strings" but have constraining facets other than the ones defined for this type, these should be defined by a customised type.

2. Dependencies

2.1 The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in RFC 2119 (reference 7.3).

2.2 The terms 'block', and 'visual' in this document are to be interpreted as described in Cascading style sheets, level 2 (reference 7.2).

2.3 References to XML, XML namespaces, XML Schema, and Unicode characters are to be interpreted according to Extensible markup language (reference 7.4), Namespaces in XML (reference 7.5), XML schema part 0 (reference 7.7), XML schema part 1 (reference 7.9), XML schema part 2 (reference 7.1), The Unicode standard: Version 3.0 (reference 7.8) respectively.

3. XML namespace

3.1 A string instance SHOULD use the namespace declaration: http://preservation.naa.gov.au/string/1.0

3.2 A string instance SHOULD use the namespace prefix: primitive

4. XML Schema

4.1 A string instance MUST conform to the following XML Schema:

<?xml version="1.0" encoding="UTF-16"?>
<xsd:schema targetNamespace="http://preservation.naa.gov.au/primitive/1.0" xmlns="http://preservation.naa.gov.au/primitive/1.0" xmlns:xsi="http://www.w3.otg/2001/XMLSchema-Instance" xmlns:xsd="http://www.w3.org/2001/XMLSchema" elementFormDefault="qualified" attributeFormDefault="unqualified">
    <xsd:element name="string">
        <xsd:simpleType>
            <xsd:restriction base="xsd:string">
                <xsd:whiteSpace value="preserve"/>
            </xsd:restriction>
        </xsd:simpleType>
    </xsd:element>
</xsd:schema>

5. Explanation of elements and attributes

5.1 string element

5.1.1 The only element in a string instance. No other element is allowed within a string instance.

XML code

<primitive:string>

Rules

The order of  characters within the string element MUST be preserved.

Whitespace within strings MUST be preserved.

Example fragment

<primitive:string>How long          is a piece of string?</primitive:string>

Attributes

nil

6. Views

6.0.1 A compliant application MUST support the requirements of the String View.

6.0.2 The String View is the default view for the string type format: when a string instance is rendered with no other arguments, it MUST be rendered in the String View.

6.1 String View

6.1.1 The String View is designed principally for visual devices.

6.1.2 The content of each string element MUST be rendered within its own block.

6.1.3 Empty string elements MUST be rendered.

6.1.4 The View MUST render numbered entities as there associated character.

6.1.5 The View MUST render the content of each element using a monospace font.

6.1.6 The View MAY wrap string element content to fit the dimensions of the view.

8. Examples

8.0.1 These examples are non-normative.

8.1 Example 1

8.1.0.1 A text file consisting of three lines with no tabs.

8.1.1 Source document

A line of text     containing the odd     tab.

8.1.2 XML markup

   <primitive:string>A line of text &#9;containing the odd &#9;tab.</primitive:string>

8.2 Example 2

8.2.0.1 A text file that uses whitespace to improve the readability of source code. Note: the '&#9;' entity in the XML markup is a XML pre-defined character reference for the tab character (Unicode character 9). A string instance may, but does not have to, use this reference.

8.2.1 Source document

//file: HelloJava.java
public class HelloJava extends javax.swing.JComponent {
 
    public static void main(String[] args) {
        javax.swing.JFrame f = new javax.swing.JFrame("HelloJava1");
        f.setSize(300,300);
        f.getContentPane().add(new HelloJava());
        f.setVisible(true);
    }
 
    public void paintComponent(java.awt.Graphics g) {
        g.drawString("Hello, Java!",125,95);
    }
}

8.2.2 XML markup

<primitive:string>
    //file: HelloJava.java</string:line>
    public class HelloJava extends javax.swing.JComponent {
    
    &#9;public static void main(String[] args) {
    &#9;&#9;javax.swing.JFrame f = new javax.swing.JFrame("HelloJava1");
    &#9;&#9;f.setSize(300,300);
    &#9;&#9;f.getContentPane().add(new HelloJava());
    &#9;&#9;f.setVisible(true);
    &#9;}
    
    &#9;public void paintComponent(java.awt.Graphics g) {
    &#9;&#9;g.drawString("Hello, Java!",125,95);
    &#9;}
    }
</primtive:string>
